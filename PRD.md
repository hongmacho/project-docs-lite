# ProjectWiki - 오버헤드 없는 프로젝트 문서화 | PRD

## 1. 서비스 개요

**프로젝트명**: ProjectWiki  
**슬로건**: 마크다운 기반의 경량 프로젝트 문서 위키 — 검색, 구성, 공유를 한 곳에서

### 문제 정의

- **Notion/Confluence의 과복잡성**: 단순 문서화만 필요한데 에디터, 데이터베이스, 템플릿, 협업 기능이 과잉
- **마크다운 파일의 불편한 검색**: 프로젝트 폴더의 `.md` 파일들이 git에는 있지만, 팀 전체가 웹에서 한눈에 보고 검색하기 어려움
- **자동화된 버전 관리 부재**: 문서 변경 이력을 추적할 수 없어 "누가 언제 뭘 바꿨는지" 알 수 없음
- **접근성 낮은 공유**: 문서를 공개/비공개하고 싶지만 단순한 매커니즘이 필요

### 해결 방안

ProjectWiki는 로컬 마크다운 폴더를 웹 기반 문서 위키로 변환:

1. **마크다운 에디터** (실시간 미리보기)
2. **자동 폴더 계층구조** (사이드바 네비게이션)
3. **전체 텍스트 검색** (SQLite FTS5)
4. **페이지 간 링크** (위키 스타일 `[[page]]`)
5. **스냅샷 기반 버전 이력** (git 커밋 대신 SQLite 변경 이력)
6. **간단한 공개/비공개 설정**

## 2. 타겟 페르소나

### 주 사용자

- **소규모 팀의 PM/리드** (5~20명): 팀 위키를 Notion 대신 간단하게 관리하고 싶음
- **개발팀 문서화 담당자**: 기술 문서, API 문서, 온보딩 가이드를 한 곳에서 관리
- **프리랜서/컨설턴트**: 프로젝트별 문서를 클라이언트와 간단하게 공유
- **개인 개발자**: 개인 프로젝트의 README, 학습 노트를 체계적으로 정리

### 사용 맥락

- 문서는 마크다운으로 이미 git에 관리 중
- 팀원들이 웹 인터페이스에서 최신 문서를 쉽게 찾고 읽길 원함
- 누가 언제 뭘 바꿨는지 변경 이력이 필요
- 빠른 로딩과 단순함을 최우선

## 3. 킬러 기능

### 전체 텍스트 검색 (FTS5)

프로젝트의 모든 문서를 페이지 제목 + 본문 + 헤딩에서 실시간 검색:

- 검색어 입력 시 즉시 결과 표시
- 검색 결과에서 매칭된 컨텍스트(주변 문장) 표시
- 저장된 검색어(북마크 검색)로 빠른 접근
- 예: "API 인증", "배포 체크리스트" 등을 몇 초 안에 찾음

## 6. 데이터 모델

### 스키마 개요

```typescript
// Documents 테이블
CREATE TABLE documents (
  id INTEGER PRIMARY KEY,
  slug TEXT UNIQUE NOT NULL,        // 'folder/page' or 'page'
  title TEXT NOT NULL,
  path TEXT NOT NULL,               // 로컬 파일 경로 (마크다운)
  content TEXT,                     // 현재 마크다운 내용
  parentId INTEGER,                 // 폴더 ID (계층구조)
  isFolder BOOLEAN DEFAULT false,
  isPublic BOOLEAN DEFAULT false,
  createdAt INTEGER,                // Unix timestamp
  updatedAt INTEGER,
  viewCount INTEGER DEFAULT 0,
  FOREIGN KEY (parentId) REFERENCES documents(id)
);

// Document Versions (변경 이력)
CREATE TABLE document_versions (
  id INTEGER PRIMARY KEY,
  documentId INTEGER NOT NULL,
  content TEXT NOT NULL,            // 해당 시점의 마크다운
  changedBy TEXT,                   // 사용자명 (향후)
  changedAt INTEGER NOT NULL,       // Unix timestamp
  changeType TEXT,                  // 'CREATE', 'UPDATE', 'DELETE'
  FOREIGN KEY (documentId) REFERENCES documents(id)
);

// FTS5 (Full-Text Search) 인덱스
CREATE VIRTUAL TABLE documents_fts USING fts5(
  title,
  content,
  path,
  content = 'documents',
  content_rowid = 'id'
);

// Search History (선택)
CREATE TABLE search_history (
  id INTEGER PRIMARY KEY,
  query TEXT NOT NULL,
  executedAt INTEGER NOT NULL
);

// Backlinks (참조 관계)
CREATE TABLE backlinks (
  id INTEGER PRIMARY KEY,
  fromDocumentId INTEGER NOT NULL,  // 링크를 포함하는 문서
  toDocumentId INTEGER NOT NULL,    // 링크 대상 문서
  mentionContext TEXT,              // 링크가 사용된 컨텍스트
  FOREIGN KEY (fromDocumentId) REFERENCES documents(id),
  FOREIGN KEY (toDocumentId) REFERENCES documents(id)
);
```

### 주요 설계 원칙

1. **SQLite FTS5**: 페이지 제목 + 본문 검색 인덱싱
2. **스냅샷 기반 버전**: `document_versions` 테이블에 각 변경 시점의 전체 마크다운 저장
3. **계층적 폴더**: `documents` 테이블의 `parentId`로 폴더 구조 표현
4. **Backlinks**: 페이지 간 참조 관계 자동 추출 및 표시

### 메인 네비게이션

- "최근 문서": 최근에 수정한 페이지 5개
- "모든 페이지": 전체 페이지 목록

- "설정": 프로젝트 설정
- "새 페이지": 페이지 생성 버튼

### 에디터 영역

- "제목을 입력하세요": 페이지 제목 placeholder
- "마크다운 형식으로 입력하세요": 본문 입력 placeholder
- "저장 중...": 자동 저장 상태

- "변경 사항 있음": 저장되지 않은 변경

- "검색 결과 N개": 검색 결과 수

- "검색 결과가 없습니다": 빈 상태 메시지

- "이 버전으로 복원": 복원 버튼

### 설정

- "프로젝트 이름": 프로젝트명 입력
- "설명": 프로젝트 설명
- "자동 저장 간격": 초 단위 입력
- "로컬 마크다운 폴더": 파일 경로 입력
- "저장": 설정 저장 버튼

## 9. 주의사항

### Git 자동 커밋 대신 SQLite 스냅샷

**원안에서 변경된 부분**:
- ~~"자동 git 커밋"~~ → SQLite의 `document_versions` 테이블로 변경 이력 관리
- 이유: git 커밋은 서버 사이드 파일시스템 접근이 필요하고, 리스크가 높음 (권한, 네트워크, 충돌 처리)
- 솔루션: 모든 변경을 SQLite에 스냅샷으로 저장하고, 변경 이력 Diff 뷰 제공

**장점**:
1. 파일시스템 의존성 제거
2. 즉각적인 저장 (git push 지연 없음)
3. 더 세밀한 버전 추적 가능

**v1.1 기능 (향후)**:
- 사용자가 명시적으로 "git에 커밋" 버튼으로 변경사항 푸시 가능
